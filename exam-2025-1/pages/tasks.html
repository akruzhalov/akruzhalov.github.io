Основное

Разработать сайт интернет-магазина, позволяющий пользователю просматривать каталог товаров, производить поиск и сортировку товаров по заданным критериям, оформлять заказы, просматривать информацию об оформленных заказах.

Вёрстка

При вёрстке можно использовать любые технологиии, изученные в течение семестра. Для каждой страницы сайта в задании представлен её вайрфрейм (wireframe), определяющий структуру и взаимное расположение основных блоков страницы. Цветовое оформление и прочие аспекты дизайна остаются на усмотение студента.

Страницы должны быть адаптированы под просмотр на мобильных устройствах. Формат отображения на мобильном устройстве по усмотрению студента, но все компоненты должны отображаться корректно, текст должен быть читабельным, не должно быть горизонтальной полосы прокрутки. Адаптивная вёрстка оценивается отдельно.

Главная страница

На главной странице должны располагаться следующие компоненты:

  • Хедер с логотипом или названием интернет-магазина, поисковой строкой, ссылками (иконками) для перехода на страницу корзины и страницу личного кабинета (ЛК).

  • Область для всплывающих уведомлений (не видна пользователю, если нет активных уведомлений).

  • Сайдбар с формой для фильтрации товаров и кнопкой «Применить». В форме должны быть следующие разделы:

    • Категория товара (checkbox),
    • Стоимость (от и до, input:number),
    • Товары со скидкой (checkbox).

  • Блок с каталогом товаров. В начале блока должен быть заголовок «Каталог» и выпадающий список для выбора порядка сортировки товаров (select). Далее должны быть карточки товаров, представленные в виде сетки. Количество карточек в одной строке настраивается, исходя из возможности использовать доступное свободное пространство. В каждой карточке присутствует изображение товара, за которым следует название, рейтинг, цена (цена со скидкой – при наличии), кнопка добавления товара в корзину. Если на странице показаны не все доступные товары, внизу блока должна быть кнопка «Загрузить ещё».

  • Футер с названием компании и контактными данными. Футер должен находиться внизу страницы вне зависимости от размера (высоты) контента страницы.

Страница корзины

На странице корзины должны располагаться следующие компоненты:

  • Хедер с логотипом или названием интернет-магазина, ссылками (иконками) для перехода на главную страницу (каталог) и страницу личного кабинета (ЛК).

  • Область для всплывающих уведомлений (не видна пользователю, если нет активных уведомлений).

  • Блок с содержимым корзины. В начале блока должен быть заголовок «Корзина». Далее должны быть карточки товаров, представленные в виде сетки. Количество карточек в одной строке настраивается, исходя из возможности использовать доступное свободное пространство. В каждой карточке присутствует изображение товара, за которым следует название, рейтинг, цена (цена со скидкой – при наличии), кнопка удаления товара из корзины. Если в корзине нет товаров, вместо сетки должен быть отображён текст «Корзина пуста. Перейдите в каталог, чтобы добавить товары.».

  • Блок с формой оформления заказа. В начале блока должен быть заголовок «Оформление заказа». За ним следует форма ввода данных, содержащая следующие поля:

    • имя (input:text),
    • email (input:email),
    • телефон (input:text),
    • согласие на получение рассылки (input:checkbox),
    • адрес доставки (input:text),
    • дата доставки (input:date),
    • врменной интервал доставки (select),
    • комментарий к заказу (textarea).

  • Футер с названием компании и контактными данными. Футер должен находиться внизу страницы вне зависимости от размера (высоты) контента страницы.

Страница личного кабинета

На странице ЛК должны располагаться следующие компоненты:

  • Хедер с логотипом или названием интернет-магазина, ссылками (иконками) для перехода на главную страницу (каталог) и страницу корзины.

  • Область для всплывающих уведомлений (не видна пользователю, если нет активных уведомлений).

  • Блок со списком оформленных заказов. В начале блока должен быть заголовок «Заказы». Далее должна быть таблица со списком заказов. Для каждого заказа должен быть отображён его порядковый номер в списке, дата и время оформления, состав (наименования товаров, перечисленные через запятую), итоговая стоимость заказа, дата и временной интервал доставки, кнопки (иконки) «Просмотр», «Редактирование», «Удаление».

  • При нажатии на кнопки «Просмотр», «Редактирование», «Удаление» пользователю должны открываться модальные окна, реализующие соответствующее действие. Примерные макеты модальных окон представлены ниже.

  • Футер с названием компании и контактными данными. Футер должен находиться внизу страницы вне зависимости от размера (высоты) контента страницы.

Скрипты для взаимодействия с API

Для реализации функциональности загрузки каталога товаров, оформления заказов и управления уже оформленными заказами вам предоставляется API (см. раздел Описание API). Вам необходимо написать скрипты на языке программирования JavaScript, реализующие взаимодействие вашей страницы с этим API посредством AJAX-запросов (можно использовать XMLHttpRequest или fetch) и позволяющий пользователю вашего сайта просматривать каталог товаров, оформлять заказы, а также просматривать существующие заказы, редактировать и удалять их.

Загрузка и отображение списка записей

  • Загрузка данных должна осуществляться посредством GET-запросов на URL, приведённые в разделе Описание API. По умолчанию в качестве ответа на этот запрос возвращаются все доступные данные в формате JSON (возвращаемый набор полей для каждой записи можно найти в этом же разделе).

  • Полученные данные должны быть размещены на странице (см. раздел Вёрстка). Набор отображаемых полей зависит от типа записи (товар, заказ) и определяется приведёнными макетами. В случае, если значение какого-либо поля слишком длинное, допускается сокращённое отображение содержимого этого поля и использование всплывающих подсказок (tooltip) для просмотра полного значения.

Редактирование заказа

  • При нажатии на кнопку редактирования заказа должно появляться модальное окно с формой для редактирования. Набор полей формы и типы элементов представлены на макете в разделе Вёрстка. Для редактирования пользователю доступны следующие поля: имя, email, телефон, адрес доставки, дата доставки, врменной интервал доставки, комментарий к заказу.

  • Заголовок модального окна – «Редактирование заказа».

  • Значения полей формы должны быть автоматически установлены в соответствии со значениями полей редактируемой записи.

  • При нажатии на кнопку «Сохранить» должен быть произведён PUT-запрос на URL, приведённый в разделе Описание API.

  • Если операция была успешно выполнена, необходимо разместить в области уведомлений сообщение об удачном завершении операции, а также обновить данные соответствующей записи в таблице. Если при выполнении операции произошла ошибка, необходимо вывести уведомление об ошибке.

  • Для оформления уведомлений необходимо использовать следующее цветовое оформление: зелёный – уведомление об успешном завершении операции, красный – уведомление об ошибке, синий – информационное уведомление. Уведомления должны автоматически исчезать по прошествии 5 секунд после их появления.

Удаление заказа

  • При нажатии на кнопку удаления заказа должно появляться модальное окно подтверждения операции удаления.

  • Окно должно содержать вопрос «Вы уверены, что хотите удалить заказ?» и кнопки «Да» и «Нет».

  • При нажатии на кнопку «Нет» происходит закрытие модального окна. При нажатии на кнопку «Да» происходит отправка DELETE-запроса на URL, приведённый в разделе Описание API. В качестве ответа сервер вернёт результат выполнения операции (см. формат ответа в разделе Описание API).

  • Если операция была успешно выполнена, необходимо удалить данные соответствующей записи из таблицы.

  • Статус выполнения операции должен быть отображён пользователю с помощью уведомления (аналогично операции редактирования).

Оформление заказа

  • Для оформления заказа пользователь должен добавить товары в корзину. Добавление происходит по нажатию на кнопку «Добавить» в карточке товара на странице каталога. Информация о добавленных товарах (их идентификаторы) хранится в localStorage.

  • При переходе на страницу корзины пользователю должны быть отображены добавленные в корзину товары. Идентификаторы товаров хранятся в localStorage, данные для отображения (название и т. д.) должны быть получены посредством API.

  • По нажатию на кнопку «Удалить» в карточке товара соответствующий товар удаляется из корзины (удаляется элемент на странице и запись о нём в localStorage).

  • Внизу формы оформления (см. макет) должна быть отображена итоговая смоимость заказа. Стоимость расчитывается как сумма стоимости всех выбранных товаров и стоимости доставки. Базовая стоимость доставки – 200 руб., в будние дни в вечерние часы (с 18:00) к стоимости доставки прибавляется 200 руб., в выходные дни – 300 руб (праздничные дни можно не учитывать). Стоимость должна динамически обновляться в зависимости от вводимых пользователем данных.

  • При нажатии на кнопку «Оформить» должен быть произведён POST-запрос на URL, приведённый в разделе Описание API. В качестве ответа сервер вернёт результат выполнения операции (см. формат ответа в разделе Описание API).

  • Статус выполнения операции должен быть отображён пользователю с помощью уведомления (см. выше).

  • При успешном оформлении заказа данные корзины в localStorage должны быть очищены, и пользователь должен быть перенаправлен на главную страницу.

Индивидуальные задания

Вариант 1 – поиск товаров с автодополнением запроса

Реализуйте функциональность поиска товаров по названию с возможностью автодополнения поискового запроса в момент его ввода.

В процессе ввода пользователем запроса ему должен быть отображён список вариантов автодополнения. Список должен отображаться непосредственно под полем ввода запроса поверх других элементов страницы (см. макет ниже). Список вариантов автодополнения формируется сервером (см. раздел Описание API). При выборе (клике) пользователем одного из вариантов, его значение должно быть записано в конец поля ввода запроса, заменяя собой последнее введённое пользователем слово, если оно было введено не полностью. Выпадающий список скрывается при выборе какого-либо из вариантов.

Поиск записей осуществляется по нажатию на кнопку «Найти». Для этого формируется запрос к серверу с указанием параметра query, значение которого должно соответствовать введённому запросу. Сервер вернёт список найденных товаров в формате JSON, которые должны быть отображены в каталоге. Если список пустой, вместо карточек товаров пользователю должен быть отображён текст «Нет товаров, соответствующих вашему запросу». При отображении результатов поиска должен учитываться порядок сортировки, выбранный пользователем.

Вариант 2 – частичное (постраничное) отображение каталога

Реализуйте функциональность частичного (постраничного) отображения каталога.

Для загрузки товаров с сервера по частям, вы можете передать в запросе параметр page с указанием номера страницы и параметр per_page для указания количества товаров на странице (по умолчанию – 10). При этом сервер пришлёт вам данные в следующем формате:

{
    "_pagination": {
        "current_page": 1,
        "per_page": 10,
        "total_count": 97
    },
    "goods": [
        {
            "actual_price": 625,
            "created_at": "2025-01-06T23:17:34",
            "discount_price": 355,
            "id": 87,
            "image_url": "https://edu.std-900.ist.mospolytech.ru/exam-2024-1/api/images/71v+9VoZYvL._AC_UL320_.jpg",
            "main_category": "sports & fitness",
            "name": "Leosportz Weight Lifting Straps with Wrist Support - Wrist Straps for Weightlifting - Superior Deadlift Straps and Workout...",
            "rating": 3.2,
            "sub_category": "Fitness Accessories"
        }
    ]
}

По ключу "_pagination" вам будет доступна информация о номере текущей страницы (current_page), количестве товаров на страницу (per_page) и общем количестве товаров (total_count). Эта информация необходима, в частности, для рассчёта общего количества страниц. По ключу "goods" сервер пришлёт вам список товаров, соответствующих текущей странице.

При открытии страницы каталога пользователю должны быть отображены товары первой страницы. Если товаров в каталоге больше, чем было отображено, пользователю должна быть доступна кнопка "Загрузить ещё" (см. макет), по нажатию на которую с сервера загружаются данные следующей страницы и отображаются в каталоге (не заменяя собой товары, загруженные ранее). Как только пользователь загрузит все доступные товары, кнопка "Загрузить ещё" должна быть скрыта.

Допускается вариант реализации через переключение страниц, при котором в каталоге отображаются товары только текущей страницы. В этом случае вместо кнопки "Загрузить ещё" нужно расположить панель навигации с кнопками для переключения страниц. Кнопка, соответствующая текущей странице, должна быть выделена.

Для сохранения выбранного пользователем порядка сортировки вы можете передать в запросе на сервер параметр sort_order с одним из следующих значений:

  • "rating_asc" – по возрастанию рейтинга,

  • "rating_desc" – по убыванию рейтинга,

  • "price_asc" – по возрастанию цены,

  • "price_desc" – по убыванию цены.

Вариант 3 – фильтрация товаров по заданным критериям

Реализуйте возможность фильтрации товров по категории, диапазону стоимости и наличию скидки.

Пользователь может выбрать критерии фильтрации в форме в боковой панели главной страницы (см. макет в разделе Вёрстка). Список доступных для выбора категорий товаров (main_category) динамически формируется на основе всех имеющихся в каталоге товаров.

После выбора пользователем критериев фильтрации, по нажатию на кнопку «Применить» должна быть произведена фильтрация товаров в каталоге – пользователю должны быть отображены только товары, соответствующие выбранным критериям. Порядок отображения должен учитывать выбранный пользователем порядок сортировки. Если пользователь не заполнил какое-либо поле формы – оно не учитывается при фильтрации. Наример, если не заполнено поле стоимости «от», то нижний порог ценового диапазона не должен учитываться. Если не заполнено ни одно поле – пользователю отображается весь каталог.

При фильтрации по стоимости должна учитываться цена товара со скидкой (discount_price), при отсутствии скидки – обычная цена (actual_price).

Авторизация

Для получения доступа к API необходимо пройти процедуру авторизации. Для авторизации нужно указать в качестве параметра запроса api_key значение уникального ключа, который выдаётся каждому пользователю. Ключ представляет собой идентификатор UUIDv4, который является случайным 16-байтным номером (например, 123e4567-e89b-12d3-a456-426655440000).

Для получения ключа необходимо перейти по ссылке, которая выложена в разделе с экзаменационным проектом в СДО Московского Политеха.

Обратите внимание, что параметр api_key всегда передаётся в строке запроса.

Пользователь может просматривать, редактировать и удалять только свои заказы. В один момент времени в базе данных может быть не более 10 заказов, созданных одним и тем же пользователем.

Если пользователь попробует совершить действие, не пройдя авторизацию, в качестве ответа на его запрос придёт сообщение 

{'error': 'Для получения доступа к API необходимо пройти процедуру авторизации. Для этого нужно передать в запросе персональный API Key.'}

При передаче параметров в POST- и PUT-запросах данные должны передаваться в теле запроса в формате application/json (нужно вручную сериализовать данные и установить значение заголовка Content-Type) или multipart/form-data (достаточно при отправке данных формы воспользоваться объектом FormData).
